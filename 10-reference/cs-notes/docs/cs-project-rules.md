# CS Project Rules (cs-project-rules.md)

이 문서는 🌼 **Blooming CS Garden** 프로젝트 전용 규칙입니다.  
어떤 채팅방에서 작업하더라도 **동일한 기준**과 **동일한 문서 퀄리티**가 유지되도록 설계된 “절대 가이드라인”입니다.

---

# 1. 전체 프로젝트 구조

## 📁 최상위 디렉토리 구조
```
/cs
 ├── os/
 ├── network/
 ├── database/
 ├── essentials/
 ├── patterns/
 ├── docs/
 └── README.md
```

각 폴더는 “주제별”로 구성하며, 모든 Day 문서는 해당 주제 폴더 내부에 생성된다.

---

# 2. Day 기반 파일 네이밍 규칙 (절대 고정)

각 Day에는 **반드시 2개의 문서**가 1세트로 생성된다.

```
DayNN-주제명-concept.md
DayNN-주제명-questions.md
```

예시:
```
Day01-os-intro-concept.md
Day01-os-intro-questions.md
Day09-network-tcpudp-concept.md
Day09-network-tcpudp-questions.md
```

**규칙 요약**
- Day 번호는 2자리(NN)
- 파일명은 소문자 + 하이픈(-)
- concept = 교재급 개념 문서
- questions = 면접 말하기 버전 질문 문서
- Day 범위를 넘는 개념/질문은 절대 포함 금지 (SRP 원칙)

---

# 3. 개념 문서(CONCEPT) 작성 규칙 (교재급 깊이)

concept 문서는 **아주 자세하고 친절한 설명**, **비유**, **구조**, **실무 연결성**을 포함한  
“교재급 문서”로 작성한다.

## ✔ Concept 문서 구조 (고정 템플릿)

1. 이 개념을 왜 배워야 하는가? (문제의식)
2. 비유로 먼저 이해하기 (쉬운 비유 2~3개)
3. 개념 정의 (공식적 정의)
4. 구조로 이해하기 (계층·흐름·도식 묘사)
5. 동작 원리 (단계별 상세 설명)
6. 실제 컴퓨터 내부에서 일어나는 일
7. 백엔드 실무에서 왜 중요한가?
8. 자주 하는 오해 vs 진실
9. **핵심 원리·구조·키워드 10줄 요약**

## ✔ Concept 문서 원칙

- 설명은 **아주 자세하게, 친절하게**
- 비유를 통해 직관적 이해 제공
- 개념 → 구조 → 동작 → 실무 → 요약 순서 유지
- **질문/답변 내용 금지** (SRP 책임 분리 원칙)
- Day 범위 외 개념 포함 금지  
  (예: Day01에 프로세스/스레드 내용을 추가하지 않음)

---

# 4. 질문 문서(QUESTIONS) 작성 규칙 (면접 말하기 버전)

questions 문서는 “실제 면접장에서 입으로 바로 말하는 스타일”로 작성한다.

## ✔ 질문 카테고리 (6종류 반드시 포함)

- 기본 질문 (Basic)
- 중급 질문 (Intermediate)
- 심화 질문 (Advanced)
- 꼬리질문 (Follow-up)
- 실무 질문 (Practical)
- 비교 질문 (Comparison)

## ✔ 답변 스타일 (말하기 형태 A+ 완전 모범답안)

모든 질문의 답변은 **말로 바로 자연스럽게 전달되면서도 전문적인 톤**으로 작성한다.

### 답변 구성 규칙:

1) 한 줄 핵심 요약  
2) 말하기 자연스러운 설명  
3) 이유(Why)  
4) 구조/원리 설명  
5) 실무 연결  
6) 짧은 마무리  

### 예시 톤:

- “OS는 프로그램이 하드웨어를 직접 만지지 못하게 하고 자원을 공정하게 관리하는 역할을 합니다.”
- “이 기능은 유저 모드에서 실행되면 위험하기 때문에 커널 모드에서만 수행하도록 설계돼 있습니다.”

## ✔ Questions 문서 원칙

- 실제 면접장에서 **그대로 말해도 어색하지 않은 말투**
- 기술적으로 정확하고 깊이 있는 설명 유지
- Day 범위 외 질문 포함 금지
- concept 문서 내용과 섞이지 않음

---

# 5. 책임 분리 원칙 (SRP, Single Responsibility Principle)

문서 간 역할은 절대 섞이지 않는다.

| 문서 종류 | 책임 |
|----------|------|
| concept.md | 교재급 설명, 원리/구조, 비유, 실무 흐름 |
| questions.md | 말하기 기반 A+ 면접 답안 |

❌ concept 문서에 질문 포함  
❌ questions 문서에 개념 설명 포함  
❌ 다음 Day 주제 선반영  

➡ 모든 문서는 **하나의 책임만** 가져야 함.

---

# 6. Day 생성 규칙 (고정)

각 Day는 아래 순서로 생성한다.

```
1️⃣ DayNN 개념 문서 생성 (concept.md)
2️⃣ DayNN 질문 문서 생성 (questions.md)
```

- Day 번호는 “커리큘럼 순서 기반”
- Day01 → OS Intro  
- Day02 → 프로세스/스레드  
- Day03 → CPU 스케줄링  
- … 이런 식으로 주제 단위로 분리

---

# 7. 모든 출력의 일관성 규칙

어떤 채팅방에서 요청하더라도 다음 항목들은 반드시 동일하게 유지된다.

- 문서 템플릿 구조  
- 파일명 규칙  
- 질문 종류 6개  
- 답변 말투(말하기 버전)  
- 개념 문서의 깊이와 친절함  
- SRP 기반 Day 범위 준수  
- 전체 폴더 구조  

---

# 8. 파일 생성 시 추가 규칙

- Markdown(.md) 고정  
- 다이어그램은 글로 묘사  
- 코드가 필요한 경우 최소 예시만 포함  
- 문서는 항상 독립적으로 읽어도 이해 가능해야 함  
- 설명 생략 절대 금지  
- 심화 내용도 Day 내 범위면 반드시 포함

이 규칙 문서는  
**봄이만의 평생 CS 아카이브 + 면접 대비 완전체**를 구축하기 위한 절대 기준이다.
